# 组织文档结构并接入 AI 生成流程

Docs Atlas 的优势不只是“能显示 Markdown”，而是它能稳定承接长期维护的项目文档，也能承接 AI 批量生成的结构化内容。

如果目录结构清晰，左侧目录、专题入口、搜索结果和阅读顺序都会更自然。

## 推荐的目录结构

最稳妥的方式是按专题组织：

```text
backend/
├── README.md
├── 01-architecture.md
├── 02-domain-model.md
├── 03-api-design.md
└── assets/
    └── architecture.png
```

建议遵循这些规则：

- 每个专题目录有一个 `README.md`
- 文件名尽量带编号，方便排序稳定
- 每篇文档使用首个 `# 一级标题`
- 正文主要使用 `##` 和 `###`
- 代码块尽量带语言标识

## 为什么这套规则重要

Docs Atlas 的很多体验都依赖这套结构：

- 左侧目录树更容易理解
- `README.md` 可以作为专题入口页
- 排序更稳定
- 搜索结果更容易分辨上下文
- 同一专题内前后阅读更自然

## 当前编辑器适合做什么

桌面端文档编辑器更适合承担这些工作：

- 修改 AI 生成的初稿
- 补充设计说明
- 调整 Markdown 结构
- 修复链接、图片、表格和代码块
- 持续维护本地项目文档

它不是重型协作编辑器，但非常适合作为本机文档维护入口。

## 推荐的 AI 文档生成要求

如果你打算让 AI 直接生成一组可接入的 Markdown 文档，建议把约束写清楚。

示例提示词：

```text
请为主题“FastAPI 项目设计文档”生成一组适合 Docs Atlas 的 Markdown 文档。

要求：
1. 输出一个专题目录
2. 目录下必须包含 README.md
3. 再输出 5 篇正文文档
4. 每篇文档必须是完整正文，不要只给提纲
5. 每篇文档以 # 一级标题开头
6. 正文主要使用 ## 和 ###
7. 代码块必须带语言标识
8. 如果涉及图片，请为图片预留 ./assets/ 相对路径
9. 如果涉及流程说明，请优先输出 Mermaid
```

## 推荐工作流

建议按这个顺序使用：

1. 先定义专题和目录边界
2. 让 AI 生成一组 Markdown 初稿
3. 人工筛选、补充和修正
4. 放入某个文档仓库的文档源目录
5. 在 Docs Atlas 桌面端继续编辑和保存
6. 需要分享时再用 Web 端静态构建

## Mermaid 和图片

如果文档里有流程图或架构图，当前更推荐两种方式：

- 流程图：优先使用 Mermaid
- 截图和示意图：放在 `assets/` 目录，用相对路径引用

这样桌面端和 Web 端都更容易保持一致。
