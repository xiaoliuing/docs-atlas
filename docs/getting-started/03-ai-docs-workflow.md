# 编写适合 Docs Atlas 的文档，并接入 AI 生成流程

Docs Atlas 的优势不只是“能看 Markdown”，而是它能承接长期积累的项目文档，也能承接 AI 生成的结构化文档。

## 推荐的文档结构

最稳妥的方式是按专题组织：

```text
backend/
├── README.md
├── 01-architecture.md
├── 02-domain-model.md
└── 03-deployment.md
```

推荐规则：

- 每个专题目录必须有 `README.md`
- 正文文件尽量使用有序编号
- 每篇文档都以 `# 一级标题` 开头
- 正文主要使用 `##` 和 `###`
- 代码块带语言标识

## 为什么要坚持这套规则

因为 Docs Atlas 的目录、搜索、专题入口和阅读顺序都基于这套规则工作。

如果结构混乱，会直接影响：

- 左侧目录可读性
- 专题入口清晰度
- 文档顺序
- 搜索结果理解成本

## AI 生成内容时怎么提要求

如果你打算让 AI 直接生成一组可接入的 Markdown 文档，建议把要求说得非常明确。

最关键的不是“让 AI 生成大纲”，而是让 AI 生成“完整、可阅读、可落地”的文档目录。

## 示例提示词

```text
请为主题“FastAPI 项目设计文档”生成一组适合 Docs Atlas 的 Markdown 文档。

要求：
1. 输出一个专题目录
2. 目录下必须包含 README.md
3. 再输出 5 篇正文文档
4. 每篇文档必须有完整正文，不要只给提纲
5. 每篇文档以 # 一级标题开头
6. 正文主要使用 ## 和 ###
7. 代码块必须带语言标识
8. 内容偏设计说明、模块说明、接入说明和部署说明
```

## 推荐工作流

你可以按这个顺序使用：

1. 先定义一个专题
2. 让 AI 生成一组 Markdown
3. 人工筛选、修改和补充
4. 放入某个文档仓库的文档源目录
5. 在 Docs Atlas 桌面端里继续编辑和修订正文
6. 用 Docs Atlas 聚合阅读

## 图片和资源怎么处理

如果文档里需要图片，推荐把图片和 Markdown 放在一起维护：

```text
backend/
├── README.md
├── 01-architecture.md
└── assets/
    └── architecture.png
```

这样在桌面端和 Web 端里都更容易保持一致。
