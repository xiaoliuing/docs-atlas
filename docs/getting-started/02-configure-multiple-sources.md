# 配置文档仓库与文档源

桌面端不再依赖 `config.yaml` 作为主配置入口。

现在更推荐的方式是：直接在桌面端里配置文档仓库和文档源。

## 文档仓库和文档源是什么关系

可以这样理解：

- 文档仓库是最外层容器
- 文档源是文档仓库下的具体目录来源
- 文档源之间可以继续按分组嵌套

这套结构的意义是让你可以在一个桌面应用里，同时管理多个项目和多套知识库。

## 推荐的组织方式

例如你有三类文档：

- 当前公司的项目文档
- 个人学习笔记
- AI 生成的方案文档

那么你可以这样组织：

- 文档仓库 A：公司项目
- 文档仓库 B：个人知识库
- 文档仓库 C：AI 方案库

每个文档仓库下面再继续挂多个目录。

## 桌面端里如何添加文档源

当前桌面端支持这些方式：

- 选择本机目录
- 手动输入目录路径
- 校验目录是否存在、是否可用
- 用分组组织多个来源

如果你有多个项目目录，推荐做法不是把它们混在一个平铺列表里，而是先分组，再挂目录。

## 什么时候适合新建文档仓库

建议在下面这些场景里新建文档仓库：

- 你希望把不同客户项目隔离开
- 你希望把工作文档和个人笔记分开
- 你希望给 AI 生成内容单独做一个阅读入口
- 你希望不同文档仓库使用不同默认搜索范围

## 桌面端会帮你记住什么

文档仓库一旦建立，桌面端会持续保留你的使用状态：

- 上次打开的是哪个文档仓库
- 该文档仓库上次阅读的是哪篇文档
- 搜索范围是“全局”还是“当前文档仓库”
- 左侧目录和分组的展开状态
- 文档滚动位置

## Web 端什么时候还需要 config.yaml

如果你要使用 Web 端做静态预览或发布，`config.yaml` 仍然有效。

示例：

```yaml
docs:
  items:
    - path: ./docs
      name: local
    - name: Workspace
      items:
        - path: ../backend-docs
          name: backend
        - path: ../mobile-docs
          name: mobile
```

说明：

- `path` 支持相对路径和绝对路径
- `name` 是模块名称，也是构建命名空间
- `items` 支持递归嵌套

也就是说：

- 桌面端主要用“文档仓库配置”
- Web 端主要用 `config.yaml`
