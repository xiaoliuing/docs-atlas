# 配置文档仓库与文档源

桌面端现在不依赖 `config.yaml` 作为主入口。

更推荐的方式是：直接在应用里创建文档仓库，并在文档仓库里配置文档源。

## 文档仓库和文档源的关系

可以这样理解：

- 文档仓库是最外层容器
- 文档源是实际挂载的目录
- 文档源之间可以再通过分组继续组织

这让你可以在一个桌面应用中同时管理多个项目、多个知识域和多套 AI 文档输出目录。

## 推荐的组织方式

例如你本机有这些内容：

- 一个后端项目文档目录
- 一个前端项目文档目录
- 一个个人学习笔记目录
- 一个 AI 生成方案归档目录

可以这样组织：

- 文档仓库 A：工作项目
- 文档仓库 B：个人知识库
- 文档仓库 C：AI 文档沉淀

每个文档仓库内部再挂多个文档源，而不是把所有目录混在一层里。

## 应用里如何添加文档源

当前桌面端支持：

- 选择本机目录
- 手动输入目录路径
- 校验目录是否存在、是否可读取
- 为文档源设置显示名称
- 用分组组织多个来源
- 拖拽调整文档源顺序

如果你的文档非常多，建议先按“项目 / 领域 / 角色”分组，再挂具体路径。

## 什么时候应该新建文档仓库

下面这些场景适合新建新的文档仓库：

- 你想把工作文档和个人笔记隔离
- 你想把不同客户或不同业务线拆开
- 你想给 AI 生成内容单独做一个仓库
- 你不希望不同类型的文档混用同一套阅读历史

## 应用会记住哪些状态

文档仓库一旦建立，桌面端会持续记住你的使用状态，包括：

- 上次打开的是哪个文档仓库
- 该文档仓库上次阅读的是哪篇文档
- 阅读滚动位置
- 最近阅读记录
- 收藏状态
- 部分目录展开状态

这也是桌面端相比纯静态站更适合日常使用的原因。

## Web 端和 config.yaml 的关系

虽然桌面端不依赖 `config.yaml`，但 Web 端仍然支持用它聚合多个目录。

示例：

```yaml
docs:
  items:
    - path: ./docs
      name: local
    - name: Workspace
      items:
        - path: ../backend-docs
          name: backend
        - path: ../mobile-docs
          name: mobile
```

可以把这两套能力分开理解：

- 桌面端：在应用里配置文档仓库和文档源
- Web 端：在项目根目录用 `config.yaml` 组织构建输入
