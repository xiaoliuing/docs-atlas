# Docs Atlas Desktop Phase 2 PRD

## 1. 背景

`Docs Atlas` 第一阶段已经验证了核心价值：

- 聚合多个 Markdown 目录
- 以文档站方式统一浏览、搜索、导航
- 适合作为本机多项目设计文档的阅读入口

下一阶段不再只把它当成一个静态文档站，而是把它升级为一个可投入生产使用的桌面端产品：

- 面向个人开发者、技术负责人、AI Coding 重度用户
- 面向“本地知识库管理”和“多项目文档聚合”
- 后续可演进为带 LLM 问答能力的个人知识工作台

桌面端必须保留当前 Web 端已经证明可用的核心阅读能力，同时补齐本地软件应有的配置、索引、管理、发布和稳定性能力。

## 2. 产品目标

### 2.1 核心目标

把 `Docs Atlas` 做成一个本地优先的桌面知识库应用，让用户可以在一个程序里管理多个工作环境，并聚合不同目录下的 Markdown 文档进行阅读、搜索和后续问答。

### 2.2 用户价值

- 不再手改配置文件即可管理文档源
- 不同项目、不同客户、不同工作场景可以隔离管理
- 文档目录多、层级深时仍然可以快速定位内容
- 本地文件变化后自动更新索引，不需要重新构建站点
- 为后续接入知识问答、摘要、关联推荐打下结构基础

### 2.3 商业化目标

- 安装即用，普通用户不需要安装 Node.js
- 支持 macOS / Windows，后续补 Linux
- 支持正式版本发布、升级、日志、错误恢复
- 支持企业或个人把它作为长期知识库工具使用

## 3. 目标用户

### 3.1 核心用户

- 个人开发者
- 独立开发者
- AI Coding 用户
- 技术负责人 / 架构师
- 需要横向查看多个仓库设计文档的团队成员

### 3.2 典型场景

- 在一个应用里查看多个项目的架构设计、接口文档、开发笔记
- 为 AI 生成的 Markdown 文档提供统一的本地承载和管理入口
- 管理“工作 / 副业 / 学习 / 客户项目”多个知识环境
- 后续基于本地文档做知识检索和问答

## 4. 产品定义

### 4.1 产品定位

`Docs Atlas Desktop` 是一个本地优先的 Markdown 知识库管理器，不是在线 CMS，不是协同文档平台，也不是单纯的 Markdown 编辑器。

它更接近：

- 本地文档聚合阅读器
- 多工作环境知识库管理器
- 面向 AI 时代的个人技术文档工作台

### 4.2 核心对象模型

#### Workspace

工作环境。用于隔离不同的知识域，例如：

- 工作项目
- 个人项目
- 学习资料
- 客户交付

每个 Workspace 独立拥有：

- 名称、描述、图标
- 文档源树
- 主题和阅读偏好
- 搜索范围和排序偏好
- 后续的 LLM 提供商、知识问答配置

#### Source Node

文档源节点。支持嵌套，类型包括：

- Group：仅用于组织结构
- Folder Source：指向本地目录
- Virtual Section：后续预留，用于收藏、最近阅读、智能视图

#### Document Asset

单篇文档及其关联资源：

- Markdown 文件
- 相对图片
- 代码块
- 标题目录
- 元数据和索引信息

## 5. 第二阶段范围

### 5.1 必做范围

#### A. 桌面端应用壳

- 提供独立可安装的桌面应用
- 启动后直接进入最近工作环境
- 支持窗口级主题切换、系统主题跟随
- 支持基础菜单、关于页、设置页

#### B. 多工作环境管理

- 新建、删除、复制、重命名 Workspace
- 工作环境切换
- 导入 / 导出 Workspace 配置
- 最近打开的 Workspace 记录

#### C. 可视化文档源管理

- 在界面中添加本地文档目录
- 支持嵌套 Group 和 Source
- 支持拖拽调整层级和顺序
- 支持启用 / 禁用某个文档源
- 支持路径有效性检测、重复路径检测、名称冲突提示

#### D. 运行时索引与实时刷新

- 不再依赖构建期扫描
- 应用启动后扫描当前 Workspace 的所有文档源
- 文件新增、修改、删除后自动更新索引
- 相对图片、相对链接在桌面端正确解析

#### E. 阅读体验保留并增强

桌面端必须保留现有 Web 端核心能力：

- 首页 / 概览页
- 左侧目录导航
- Markdown 详情页
- 右侧目录大纲
- 上一篇 / 下一篇
- 全文搜索
- 代码高亮
- 图片预览
- 浅色 / 暗色主题

同时增强：

- 阅读历史
- 最近打开文档
- 固定收藏
- 文档内搜索
- 更适合大屏的三栏布局

#### F. 搜索能力升级

- Workspace 级全文搜索
- 按标题、正文、模块、来源过滤
- 搜索结果关键字高亮
- 点击结果后定位到文档命中位置
- 搜索索引持久化，避免每次启动全量重建

#### G. 设置与生产能力

- 设置页集中管理应用配置
- 本地数据目录可查看
- 导入失败 / 索引失败有明确错误提示
- 崩溃后可恢复最近 Workspace
- 提供日志导出入口

### 5.2 可选增强范围

- 多窗口阅读
- 多标签页阅读
- 收藏夹 / 智能书签
- 最近变更文档视图
- 文档源健康检查
- 自定义首页卡片
- 全局快捷键打开搜索

### 5.3 暂不纳入第二阶段

- 在线协作编辑
- 云端同步
- 权限系统
- 团队账号体系
- 复杂所见即所得编辑器
- 远程仓库直接拉取与自动认证管理

## 6. 桌面端产品形态

### 6.1 首选技术路线

建议采用 `Monorepo + Tauri 2 + Vue 3`。

原因：

- 现有 Web 端前端能力可以最大化复用
- Tauri 安装包更轻，资源占用更可控
- 文件系统、窗口、系统菜单、日志、自动更新等桌面能力更自然
- 后续可用 Rust 处理文件扫描、监听、索引和本地数据库

如果后续发现桌面端需要更重的 Node 生态桌面插件，再评估 Electron，但不建议在第二阶段一开始就选 Electron。

### 6.2 Monorepo 结论

是，建议进入第二阶段时改成 Monorepo。

原因不是“流行”，而是桌面端和 Web 端已经明确会长期共享下面这些能力：

- Markdown 解析
- 文档扫描规则
- 路由和 slug 生成规则
- 目录树模型
- 搜索命中模型
- 类型定义
- 阅读组件和样式 token

如果继续单包硬叠，会很快把桌面端逻辑、Web 端逻辑和共享逻辑混在一起，后续很难维护。

### 6.3 推荐仓库结构

```text
docs-atlas/
├── apps/
│   ├── web/                 # 现有 Web/SSG 版本
│   └── desktop/             # Tauri 桌面端
├── packages/
│   ├── docs-core/           # 扫描、解析、slug、目录树、链接与资源处理
│   ├── search-core/         # 搜索索引、命中模型、排序
│   ├── shared-types/        # Doc / Workspace / Source 等类型
│   ├── ui-core/             # 共享 Vue 组件与样式 token
│   └── config-core/         # 配置读写、校验、迁移
├── open-docs/
│   └── phase-2-desktop-prd.md
└── tooling/
    ├── scripts/
    └── configs/
```

### 6.4 迁移原则

- Web 端不做大改，不中断当前可用能力
- 先抽共享核心，再接桌面壳
- 共享逻辑优先抽到 `packages/*`
- 桌面端先实现单机可用，再补自动更新和问答

## 7. 功能需求

### 7.1 Workspace 管理

- 支持创建空 Workspace
- 支持从本地目录快速创建 Workspace
- 支持导入旧版 `config.yaml`
- 支持导出当前 Workspace 配置
- 支持设置默认启动 Workspace

### 7.2 文档源管理

- 文档源支持无限层级嵌套
- 同一 Workspace 内支持多个根来源
- 每个节点支持：
  - `name`
  - `type`
  - `path`
  - `enabled`
  - `children`
- 支持排除规则：
  - `node_modules`
  - `.git`
  - `dist`
  - 用户自定义 ignore

### 7.3 阅读与导航

- 目录树自动展开当前文档链路
- 支持目录搜索
- 文档大纲和正文联动
- 图片点击放大、缩放、拖拽、上下切换
- 文档标题、摘要、来源信息展示
- 支持复制文档路径、打开原文件目录

### 7.4 搜索

- 默认搜索当前 Workspace
- 支持切换到“全部工作环境”
- 支持按来源 / 模块 / 文件类型过滤
- 搜索结果展示来源归属、摘要、命中位置
- 支持键盘上下选择与回车打开

### 7.5 桌面集成

- 打开本地目录
- 在系统文件管理器中定位文件
- 拖拽文件夹到窗口中导入
- 应用关闭后保存布局状态
- 支持自启动为可选项

## 8. LLM 演进边界

第二阶段不要求交付完整知识问答，但必须为第三阶段预留接口。

### 8.1 第二阶段要预留的能力

- Workspace 级知识索引 ID
- 文档切片与 chunk 元数据模型
- 文档更新时间和索引版本号
- 可扩展的检索接口
- Provider 配置入口

### 8.2 第三阶段目标

- 接入本地或云端 LLM Provider
- 基于当前 Workspace 做检索增强问答
- 支持引用来源回溯
- 支持按 Workspace 隔离问答知识域
- 支持用户控制是否允许索引特定目录

## 9. 非功能要求

### 9.1 性能

- 冷启动进入最近 Workspace 小于 3 秒
- 常规 5,000 篇 Markdown 以内可用
- 搜索响应在 200ms 内返回首屏结果
- 文件变更后 2 秒内完成增量刷新

### 9.2 稳定性

- 配置损坏可回退到上一个有效版本
- 单个目录读取失败不影响整个 Workspace 打开
- 索引异常、图片缺失、链接异常要有可诊断提示

### 9.3 安全与隐私

- 默认本地运行，不上传用户文档
- LLM 接入必须显式授权
- 所有外部请求都可关闭
- 日志脱敏，不记录正文全文

### 9.4 可维护性

- Web / Desktop / Shared Core 有清晰边界
- 配置模型有版本号和迁移机制
- 关键流程有单元测试和集成测试

## 10. 发布与交付策略

### 10.1 Git 仓库与工程管理

第二阶段开始建议切换到 `pnpm workspace` Monorepo。

### 10.2 GitHub Action 策略

不建议所有推送都触发 GitHub Pages。

建议拆成三类流程：

#### Web Demo Workflow

- 只在 `apps/web/**`、共享前端包、或手动触发时运行
- 负责 GitHub Pages 示例站

#### Desktop CI Workflow

- 在桌面端代码或共享核心变更时运行
- 只做安装、测试、构建校验
- 不负责部署 Demo

#### Desktop Release Workflow

- 只在打标签或手动发布时运行
- 生成 macOS / Windows 安装包
- 上传到 GitHub Releases

这样可以避免“每次桌面端提交都走页面部署”，也能把发布责任拆干净。

## 11. 实施计划

### Phase 2A: 架构重组

- 切换到 Monorepo
- 抽离 `shared-types`
- 抽离 `docs-core`
- 抽离 `search-core`
- 让 Web 端先改成消费共享核心

### Phase 2B: 桌面壳与本地配置

- 初始化 `apps/desktop`
- 实现 Workspace 持久化
- 实现目录选择器
- 实现 Source Tree 配置 UI

### Phase 2C: 运行时索引

- 接入文件监听
- 增量更新索引
- SQLite 或等价本地存储落地
- 文档资源路径在桌面端全链路打通

### Phase 2D: 生产化完善

- 安装包
- 自动更新
- 错误日志
- 数据迁移
- 导入导出

### Phase 2E: 问答底座

- 设计 chunk、embedding、retrieval 接口
- 接入实验性问答入口
- 支持结果引用与来源跳转

## 12. 验收标准

### 12.1 MVP 验收

- 用户不安装 Node.js 也能安装和启动应用
- 用户可创建至少 3 个 Workspace
- 每个 Workspace 可独立配置多个嵌套文档源
- 新增 Markdown 后应用可自动刷新
- Web 端已有核心阅读能力在桌面端可用
- 搜索结果支持高亮并能定位正文

### 12.2 生产可用验收

- 具备安装包和升级路径
- 用户配置有持久化和迁移机制
- 异常有日志和可恢复机制
- 桌面端和 Web 端共享核心规则，不出现解析结果分叉

## 13. 结论

第二阶段不是“再做一个 Web 壳”，而是把 `Docs Atlas` 从文档站升级成真正的本地知识库产品。

建议的方向很明确：

- 工程上：改成 Monorepo
- 产品上：引入 Workspace 和可视化文档源管理
- 桌面端上：采用 Tauri 2，保留现有 Web 核心能力
- 发布上：把 Web Demo 和 Desktop Release 的 GitHub Action 拆开
- 能力上：先把本地知识库管理做扎实，再引入 LLM 问答

这个方向能兼顾你现在已有的 Web 资产，也能让项目具备继续走向正式产品的基础。
